# 内容寻址附件仓库设计

## 目标与范围

为图片、扫描 PDF、DOCX 内嵌资源和 OCR 派生结果提供统一的本地附件仓库。本批完成 Rust 文件服务、SQLite 元数据、迁移和自动化测试，并把 OCR 结果接入可恢复导入草稿；不增加通用附件管理 UI，也不实现未引用资源自动清理。

## 架构

附件二进制文件保存在应用数据目录下的 `assets/`，SQLite 只保存元数据。`AssetStore` 负责文件导入、哈希、路径生成和安全读取；`assets` repository 负责元数据查询与写入；Tauri command 只暴露“导入附件”和“读取附件信息”等业务操作。数据库主门面不承载附件 SQL。

文件按 SHA-256 内容哈希寻址，路径格式为 `assets/<前两位>/<完整哈希>.<安全扩展名>`。相同内容无论原文件名如何，只保留一份二进制文件和一条资产记录。资产 ID 使用 UUID，原始文件名仅作为展示元数据。

## 数据模型

`assets` 表包含：

- `id`：稳定 UUID；
- `sha256`：唯一内容哈希；
- `relative_path`：相对应用数据目录的受控路径；
- `original_name`：导入时文件名；
- `mime_type`：探测或调用方提供的 MIME；
- `byte_size`：文件大小；
- `created_at`：创建时间。

派生附件另外记录 `parent_asset_id`、`role`、`provider_id` 和 `model`。`role` 首版支持 `source`、`ocr_raw` 和 `ocr_markdown`，用于区分原始扫描件、OCR 原始 JSON 和标准化 Markdown。派生附件仍按内容哈希去重，但保留与来源附件和模型调用的关系。

迁移版本升级到 3。已有数据库升级时创建表和哈希唯一索引；空数据库由 schema 建表后记录迁移版本。

## 写入与读取流程

导入时先读取文件并限制单文件最大 100 MiB，再计算 SHA-256。若哈希已经存在，直接返回已有资产，不重复复制。否则根据 MIME/原扩展名选择白名单扩展名，写入同目录临时文件后原子重命名，并在 SQLite 写入元数据。若数据库写入失败，删除本次新建且尚未被引用的文件。

读取时只接受资产 ID，不接受调用方传入任意路径。服务从数据库取得 `relative_path`，将其拼接到应用数据根目录并规范化；解析后的路径必须仍位于根目录内，否则拒绝。不存在、目录、符号链接或大小与记录不符时返回明确错误。

## OCR 到题库的数据流

OCR 识别开始时，原始图片或扫描 PDF 作为 `source` 附件保存。服务响应后，原始 JSON 和标准化 Markdown 分别作为 `ocr_raw`、`ocr_markdown` 派生附件保存，并记录来源附件、Provider、模型和生成时间。这样用户重新打开草稿时不需要再次消耗模型额度。

标准化 Markdown 继续进入现有 `DocumentBlock -> QuestionDraft` 管线。切题结果保存到可恢复的导入草稿，不自动写入正式题库。校正完成后，用户必须选择：

1. 创建新题库并导入，作为默认选项；
2. 从已有题库中选择一个并追加题目。

用户也可以只保存草稿并稍后继续。原始附件、OCR 派生结果、导入草稿和正式题库保持独立生命周期。

## 安全与错误处理

- 不信任原文件名，不允许其参与目标路径；
- 只允许白名单扩展名，未知格式使用 `.bin`；
- 拒绝大于 100 MiB 的文件；
- 禁止 `..`、绝对路径和符号链接逃逸；
- 写入采用临时文件加原子重命名，避免中断留下半文件；
- 哈希冲突由唯一索引和写入后回读共同处理；
- 错误信息不包含附件内容。

## 测试

Rust 测试覆盖：同内容去重、不同内容独立存储、100 MiB 限制、受控相对路径、路径越界拒绝、迁移版本 3、派生关系、元数据往返和文件内容往返。前端测试覆盖 OCR 草稿恢复、新建题库默认值、追加到已有题库和未选择目标题库时禁止导入。最终运行 Rust 格式/编译/测试、前端测试与生产构建、`git diff --check`。
